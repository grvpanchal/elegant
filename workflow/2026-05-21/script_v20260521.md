## [SEGMENT 1] — 0:15 → 1:10
### #1 — Nuxt MCP Toolkit ships MCP apps

Number one — Nuxt's MCP toolkit just landed a feature called MCP apps. Here's what changed.

Until this week, when an LLM called a tool — say, "fetch order status" — it got back plain text or JSON, and your chat UI had to render it. With `defineMcpApp`, the tool itself returns rendered HTML that drops inline into the agent's interface. That means a Nuxt component, hydrated, interactive, sitting inside the conversation.

So what? This is the first credible pattern for shipping micro-frontends straight into the agent surface. If your roadmap has any AI-embedded internal tooling on it, this is the shape it's about to take. Watch this space.

> Source: https://vercel.com/changelog/nuxt-mcp-toolkit-mcp-apps

## [SEGMENT 2] — 1:10 → 2:05
### #2 — Vite ships 8.0.13 and 8.0.14 patches

Number two — Vite cut two patches this week. 8.0.13 on May 14, 8.0.14 on May 21, plus plugin-legacy 8.0.2 and create-vite 9.0.7. No breaking changes, no new features — these are stabilisation passes on the Rolldown line.

And that's the signal. Vite 8 ships with Rolldown as its single Rust-based bundler, and we're still in the window where regressions surface in patch releases. If you've held off upgrading from Vite 7, the v8 surface is firming up — but I'd give it another two patches before I'd bet a production migration on it.

Watch the changelog, not the headline.

> Source: https://github.com/vitejs/vite/releases/tag/v8.0.14

## [SEGMENT 3] — 2:05 → 3:00
### #3 — Grok Build 0.1 on Vercel AI Gateway

Number three — Vercel quietly added a model called Grok Build 0.1 to AI Gateway. It's beta, early-access, and the spec sheet is explicit — this isn't a general chat model. It's tuned for agentic coding workflows.

Why does that matter? Because the AI Gateway pattern — multi-vendor, behind a single provider-agnostic interface — is becoming the right architectural call. You don't pick one model, you route through a gateway and A/B the coding-tuned ones against the general ones.

Grok Build is the third or fourth coding-specialised model to drop in two months. The category is real now. If you're still hard-coding a single provider in your stack, this is the cue to refactor.

> Source: https://vercel.com/changelog/grok-build-0-1-now-available-on-vercel-ai-gateway

## [SEGMENT 6] — 4:40 → 5:35
### #6 — Formal verification gates for agent loops

Number six — there's an essay making rounds on Hacker News called "Structural Backpressure Beats Smarter Agents." One hundred and thirty-five points, thirty-plus comments. Worth the read.

The argument — you can't make agents reliable by waiting for smarter models. You make them reliable by adding verification gates in the loop. Type checks, tests, property checks — the agent has to pass each gate before iterating.

So what? If your team is shipping agent-built code, this is the architectural framing you should anchor your PR-gate policy on. It moves the conversation from "is the model good enough" to "is our pipeline strict enough." Which is the right question.

> Source: https://reubenbrooks.dev/blog/structural-backpressure-beats-smarter-agents/

## [SEGMENT 7] — 5:35 → 6:30
### #7 — codegraph — pre-indexed knowledge graph for coding agents

Number seven — there's a project called codegraph trending hard on GitHub this week. Six-thousand-seven-hundred stars in seven days.

What it does — it builds a pre-indexed knowledge graph of your codebase and exposes it to AI coding agents — Claude Code, Codex, Cursor. The bet is that context engineering at the repo level is becoming its own discipline. You don't just point an agent at a folder anymore — you give it a queryable map.

So what? For any team running agents on a large monolith, this is the layer between the IDE and the codebase that's going to standardise. Worth watching the API shape, even if you don't adopt it.

> Source: https://github.com/colbymchenry/codegraph

## [SEGMENT 8] — 6:30 → 7:25
### #8 — agentmemory — persistent memory for coding agents

Number eight — also trending, a TypeScript library called agentmemory. Eight thousand stars in a week. It's a persistent memory layer for AI coding agents, benchmarked against real workflows.

The problem it solves is the one everyone has — your agent forgets everything between sessions. Every conversation starts from zero.

So what? Stateful agents are about to become the default. And picking your memory layer is a one-way architectural decision — once your agents depend on it, you can't easily swap. This isn't a "try it in a side project" situation. If your team is moving toward persistent agents, the field of options matters. Know what's out there.

> Source: https://github.com/rohitg00/agentmemory

### #9 — 12-factor-agents crosses 21k stars

Number nine — humanlayer's 12-factor-agents framework crossed twenty-one thousand stars this week. The name is a deliberate echo of the Heroku 12-factor app manifesto, and it's the same idea — a list of design principles for production-grade LLM apps. Statelessness, explicit context boundaries, config-as-code, separation of concerns.

Look — most teams ask the wrong question. They ask "which framework should we use for agents." The right question is "what should the shape of our agent code be." This document answers the second one.

It's a reference architecture and a vocabulary. Use it to push back on bad designs in your team.

> Source: https://github.com/humanlayer/12-factor-agents

### #10 — InsForge — open-source Heroku for coding agents

Number ten — a Show HN post that hit sixty-one points — InsForge, an open-source Heroku for coding agents. It's a control plane for running Claude Code, Codex, and similar agents on managed sandboxes, with secret injection and isolated environments.

### #11 — Haystack — review-worthy PR filtering

Number eleven — another Show HN — Haystack. A tool that filters incoming pull requests and surfaces only the ones that actually need human review. Forty-four points.

The premise is simple — when half your PRs come from agents, your review queue becomes noise. Haystack triages — it flags the diffs where human judgement actually moves the needle and quietly drops the rest.

So what? Review load is about to change shape. If your team is running agents at scale, tooling that re-prioritises your review queue is going to be table stakes within six months. Get ahead of it now.

### #13 — agent-skills — signed registry for agent capabilities

Number thirteen — a TypeScript project called agent-skills. Two thousand stars this week. It's a secure registry for publishing signed, versioned agent skills using the open Agent Skills format.

The interesting angle is governance. Once your org publishes internal skills, you need the same discipline you have on dependencies — signing, versioning, vulnerability scanning. Architects, this is your domain — not the prompt engineers'.
